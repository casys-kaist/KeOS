1//! This modules explains debugging tips for KeOS.
2//!
3//! ## Selectively run tests
4//!
5//! In KeOS, you can run one or more specific test cases by passing their names
6//! as arguments to the test runner. For example:
7//!
8//! ```bash
9//! $ cargo run -- syscall::pipe_normal syscall::pipe_partial
10//! ```
11//!
12//! This command runs exactly the listed test cases, `syscall::pipe_normal` and
13//! `syscall::pipe_partial`. You may specify a single test case or multiple test
14//! cases, depending on your needs.
15//!
16//!
17//! ## Debugging with `print!`
18//! `print!` is a simple yet effective debugging tool in KeOS. It allows you to
19//! output messages directly to the console, helping you trace the execution.
20//! Use `print!` generously while debugging.
21//!
22//! ### Example: Workflow to debug why `keos_project2::userprog::arg_parse` has no output.
23//!
24//! There are two possibilities:
25//! 1. The user program executed **but never reached the print**.
26//!     * This implies the program’s initial state was wrong.
27//!     * In the grader directory, you can check that `arg_parse` is launched
28//!       via `run_elf_with_arg`.
29//!     * Add logging around that path to dump the final state before entering
30//!       user mode:
31//!         * Print a summary of `mm_struct` and the register set (`regs`).
32//!         * Because the existing `mm_struct`/page-table walk helpers may only
33//!           return a **final PTE**, write small debug helpers to dump
34//!           **intermediate** page-table levels and the final PTE (include
35//!           addresses and flags).
36//!
37//! 2. The user program executed, reached the print, **but failed to print**.
38//!    * User–kernel interactions go through the syscall interface, so if it
39//!      executed at all, it likely issued a syscall.
40//!         * Add a log in `Process::syscall` to print the syscall
41//!           **number/args** and the **return_val**.
42//!         * If you see this log, the program executed, made a syscall, and the
43//!           kernel handled it without panicking.
44//!         * Check whether `return_val` is what you expect. If not, dig deeper:
45//!    * Additional symptom: syscall number is `Write`, but the result is
46//!      `KernelError::BadAddress`
47//!         * Add a log inside `Write` to find where `KernelError::BadAddress`
48//!           originates.
49//!         * If `UserU8SliceRO::get` returns the error, the kernel thinks the
50//!           user buffer address is invalid.
51//!         * If you believe the address should be valid, continue down the call
52//!           chain: **`UserU8SliceRO::get`** → **`Task::access_ok`** →
53//!           **`MmStruct::access_ok`** (your implementation). Review these
54//!           carefully.
55//!
56//!
57//!
58//! ### Trust, but verify, your debug prints
59//! * Your logging may call code you wrote, which might itself be buggy. That
60//!   can mislead you.
61//!   * For example, if you use `get_user_page` (also your code) to dump user
62//!     memory. If it’s wrong, you could read the wrong memory and still “see”
63//!     plausible contents. In this case, use GDB to inspect the real memory
64//!     state during execution. Treat that as ground truth.
65//!

68//! ## Debugging with GDB
69//!
70//! KeOS supports debugging with **GDB** and **QEMU**. This section provides
71//! step-by-step instructions on how to set up and use GDB for effective
72//! debugging.
73//!
74//! ### Running GDB
75//!
76//! To launch KeOS in debug mode, run the following command **inside each grader
77//! directory**:
78//! ```bash
79//! $ GDB=1 cargo run <TESTCASE>
80//! ```
81//! You must specify a single test case to debug with a GDB.
82//!
83//! This starts **QEMU** and waits for a GDB connection on TCP **port 1234**.
84//! A `.gdbinit` script will also be generated to automate the debugging setup.
85//!
86//! In a **separate terminal**, start GDB using:
87//! ```bash
88//! $ rust-gdb keos_kernel
89//! ```
90//!
91//! We recommend to use `rust-gdb`, as it provides better support for
92//! Rust-specific data structures and improves debugging readability.
93//!
94//! #### One-time setup
95//!
96//! Before using `rust-gdb`, you may need to modify your **`~/.gdbinit`** file
97//! to allow script execution. Add the following line:
98//! ```bash
99//! set auto-load safe-path /
100//! ```
101//!
102//! After launching `rust-gdb`, the execution will halt at the startup stage,
103//! showing output similar to this:
104//! ```bash
105//! $ rust-gdb
106//! warning: No executable has been specified and target does not support
107//! determining executable automatically. Try using the "file" command.
108//! 0x000000000000fff0 in ?? ()
109//! (gdb)
110//! ```
111//!
112//! Now, you can continue execution by typing:
113//!
114//! ### Inspect Each Core
115//!
116//! In **QEMU**, each **CPU core** is treated as a **separate thread**.
117//! When debugging multi-core execution, be aware that **some cores may panic
118//! while others continue running.**
119//!
120//! To inspect all active cores, use:
121//! ```bash
122//! (gdb) info threads
123//! ```
124//!
125//! This will display the state of each thread, including which CPU core it
126//! belongs to and its current stack frame. For example:
127//! ```text
128//! (gdb) info threads
129//! Id   Target Id         Frame
130//! * 1    Thread 1 (CPU#0 [running]) 0x000000000000fff0 in ?? ()
131//! 2    Thread 2 (CPU#1 [running]) 0x000000000000fff0 in ?? ()
132//! 3    Thread 3 (CPU#2 [running]) 0x000000000000fff0 in ?? ()
133//! 4    Thread 4 (CPU#3 [running]) 0x000000000000fff0 in ?? ()
134//! ```
135//!
136//! To switch to a specific **CPU core (thread)**, use:
137//! ```bash
138//! (gdb) thread {thread_id}
139//! ```
140//!
141//! This allows you to inspect registers, call stacks, and execution state
142//! per core.
143//!
144//! ### Analyzing Execution State
145//!
146//! #### Viewing the Call Stack (Backtrace)
147//!
148//! Use `backtrace` (or `bt`) to display the **call stack** of the current
149//! thread:
150//! ```bash
151//! (gdb) bt
152//! ```
153//!
154//! Each function call in the stack is represented as a **frame**.
155//! To switch to a specific frame, use:
156//! ```bash
157//! (gdb) frame {frame_id}
158//! ```
159//!
160//! Once inside a frame, you can inspect variables:
161//! ```bash
162//! (gdb) info args
163//! (gdb) info locals
164//! (gdb) i r
165//! ```
166//!
167//! **Debugging a Panic:**
168//! If you encounter a **kernel panic** during a test, use:
169//! 1. `info threads` to locate the crashing core
170//! 2. `bt` to examine the backtrace
171//! 3. `frame {frame_id}` to inspect function parameters
172//!
173//! ### Setting Breakpoints
174//!
175//! Breakpoints help stop execution at specific points. However, in **multi-core
176//! debugging**, regular breakpoints may not always work correctly.
177//! Instead, use **hardware breakpoints**:
178//! ```bash
179//! (gdb) hb * {address_of_breakpoint}
180//! ```
181//!
182//! To view the source code that the current CPU is executing, use:
183//! ```bash
184//! (gdb) layout asm
185//! (gdb) layout src
186//! ```
187//!
188//! #### Examples
189//!
190//! Here are some examples of how to set breakpoints in GDB:
191//! ```text
192//! (gdb) hbreak function_name  # Example: hbreak keos::fs::Directory::open
193//! (gdb) hbreak *address       # Example: hbreak *0x1000
194//! (gdb) hbreak (file:)line    # Example: hbreak syscall.rs:164
195//! ```
196//!
197//! #### Example 1
198//!
199//! To debug the `syscall::read_normal` test case in project 1, and set a
200//! breakpoint at `syscall.rs:150`, use:
201//! ```bash
202//! (gdb) hbreak syscall.rs:150
203//! ```
204//!
205//! Alternatively, you can set a breakpoint by the test case's name:
206//! ```bash
207//! (gdb) hbreak project1_grader::syscall::read_normal
208//! ```
209//!
210//! You can even set a breakpoint on the closure entry, for instance, to set a
211//! on a closure of `sync::semaphore::sema_0` test case in project 3:
212//! ```bash
213//! (gdb) hbreak project3_grader::sync::semaphore::{{closure}}
214//! ```
215//!
216//! To limit debugging to one core, use `thread apply`:
217//! ```bash
218//! (gdb) thread apply 1 hbreak syscall.rs:150
219//! (gdb) c
220//! ```
221//!
222//! #### Example 2
223//!
224//! To stop at a breakpoint only when a specific condition is met (e.g., when a
225//! parameter is `0xcafe0000`), use:
226//! ```bash
227//! (gdb) hbreak walk if va.__0 == 0xcafe0000
228//! ```
229//!
230//! This approach allows you to focus on specific conditions and skip over
231//! unrelated calls.
