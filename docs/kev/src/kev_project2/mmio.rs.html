1//! MMIO based simple printer device.
2//!
3//! ## Background
4//! Memory-mapped I/O uses the same address space for both main memory and I/O
5//! devices. Unlike PIO, which uses dedicated instructions, in MMIO, one can
6//! think of the memory and register of I/O devices as if they are mapped in a
7//! specific address of main memory. This method allows for the convenient and
8//! fast definition of I/O device behavior and concise implementation of the CPU
9//! and I/O devices.
10//!
11//! In memory-mapped I/O, the same memory address space is used for accessing
12//! both main memory (RAM) and I/O devices. Commands for accessing main memory
13//! (such as load and store) are also used to access devices, reading from and
14//! writing to memory instead. For instance, if there exists memory-mapped I/O
15//! (MMIO) mapping between a console device and CPU at address 0x00ff, writing
16//! 'a' to the 0x00ff will print a letter 'a' to the console device.
17//!
18//! The specific area of memory for these operations can be temporarily agreed
19//! upon between a CPU and device or it can be a permanently assigned area.
20//! As an example, modern PCIe devices negotiate the locations of the MMIO
21//! sections with a CPU by using a base address register (BAR) in the
22//! initialization process.
23//!
24//! On modern CPUs, MMIO is performed by the memory management unit (MMU), which
25//! is a hardware component responsible for managing memory access.
26//! When a program running on the CPU performs an I/O operation on a
27//! memory-mapped location, the MMU intercepts the memory access and sends it to
28//! the appropriate hardware device, which performs the requested operation.
29//! Each I/O device monitors the CPU's address bus and when the memory is
30//! accessed, the device executes the command and writes the result to a
31//! specific memory location or performs the command.
32//!
33//! Memory-mapped I/O is generally faster than Programmed I/O (PIO) because it
34//! avoids the overhead of the processor having to manage the I/O operations
35//! directly. In PIO, the CPU should execute IN and OUT operations to
36//! communicate with peripheral devices. Which causes delays and limits overall
37//! system performance. In contrast, MMIO allows I/O devices to be directly
38//! mapped to memory and managed by MMU. This makes I/O operations can be
39//! performed quickly and efficiently, without requiring the CPU to spend a lot
40//! of time managing the I/O operations directly.
41//!
42//!
43//! ## Tasks
44//! In this project, you are requested to implement simple virtual MMIO control
45//! for the purpose of the printing text on the host console. In our MMIO
46//! PrinterDev specifications, we dictate the usage of 0xcafe0000 guest physical
47//! address for the buffer, which contains an array of utf8 strings. The length
48//! of the array can be founded at the 0xcafe0008, while the doorbell is located
49//! at 0xcafe0010. The doorbell in MMIO typically refers to a deginated memory
50//! location used to trigger device operations. The PrinterDev virtual device
51//! lauches an operation when a write occurs to the doorbell address, and
52//! subsequently fetches the address and length of the string buffer.
53//! It then parses the string data from the buffer and outputs given text to the
54//! host console. In summary:
55//! * 0xcafe0000: Guest physical address for the utf8 string buffer
56//! * 0xcafe0008: Length of the utf8 string buffer
57//! * 0xcafe0010: The doorbell which notifies VMM to print the registered string
58//!   to the console
59//!
60//! Your device SHOULD parses the string data from the buffer and outputs the
61//! given text to the teletype device. You can translate a raw pointer to slice
62//! by using [`core::slice::from_raw_parts`].
63//!
64//! [`core::slice::from_utf8`]: https://doc.rust-lang.org/beta/core/str/fn.from_utf8.html
65//! [`core::slice::from_raw_parts`]: https://doc.rust-lang.org/std/slice/fn.from_raw_parts.html
66
