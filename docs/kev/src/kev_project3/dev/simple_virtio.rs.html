1//! Simple VirtIO Block
2//!
3//! ## Background
4//! ### Simple VirtIO Block device specification
5//! ### 1. Introduction
6//! This describes the specifications of the "Simple VirtIO Block" device.
7//! The purpose of this documentations is to provide the basic and concrete
8//! informations for building svirtb-device and svirtb-driver.
9//!
10//! #### 1.1 Terminology
11//! The keywords "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD",
12//! "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to
13//! be interpreted as described in [RFC2119](http://www.ietf.org/rfc/rfc2119.txt).
14//!
15//! #### 1.2 Structure Specification
16//! Device and In-memory structure layouts are documented using the C struct
17//! syntax. All structure MUST not have any additional padding.
18//!
19//! For the integer data types, the following conventions are used:
20//! * u8, u16, u32, u64: An unsigned integer of the specified length in bits.
21//! * le8, le16, le32, le64: An unsigned integer of the specified length in
22//!   bits, in little-endian byte order.
23//!
24//! ### 2. Basic Facilities of a Simple VirtIO Block
25//! A simple virtIO block is discovered and identified by a mmio struct
26//! [`VirtIoMmioHeader`]. Mmio field (header) consists of the following parts:
27//! ```C
28//! struct VirtIoMmioHeader {
29//!     le32 status;
30//!     le32 queue_size;
31//!     le32 queue_addr_hi;
32//!     le32 queue_addr_lo;
33//!     le32 queue_head;
34//!     le32 queue_tail;
35//! }
36//! ```
37//! * status: Status of the device
38//! * queue_size: Size of the virtqueue
39//! * queue_addr_hi: Upper 32bit of the virtqueue physical address
40//! * queue_addr_lo: Lower 32bit of the virtqueue physical address
41//! * queue_head: Head of the ring buffer. Driver update the tail of the queue.
42//!   Device must not update the field.
43//! * queue_tail: Tail of the ring buffer. Device update the tail of the queue.
44//!   Driver must not update the field.
45//!
46//! #### 2.1 Device status field
47//! During device initialization by a driver, the driver follows the sequence of
48//! steps specified in [`3`](#3-device-initialization). The device status field
49//! provides a simple low-level indication of the completed steps of this
50//! sequence. It's most useful to imagine it hooked up to traffic lights on the
51//! console indicating the status of each device. The following status code are
52//! defined (listed below in the order in which they would be typically set):
53//! ```C
54//! le32 MAGIC = 0x74726976
55//! le32 DRIVER_OK = 0
56//! le32 READY = 1
57//! le32 RESET = 2
58//! ```
59//! * MAGIC : Indicates that the region of the mmio is used for svirtb. Device
60//!   sets the fields when initialization.
61//! * DRIVER_OK : Indicates that the driver is ready to initiate the svirtb.
62//! * READY : Indicates that the device is ready to be used by the svirtb device
63//!   driver. From this points, svirtb device is activated.
64//! * RESET : Reset the svirtb connection and start over.
65//!
66//! #### 2.1.1 Driver Requirements: Device Status Field
67//! The driver MUST update device status, setting bits to indicate the steps of
68//! the driver initialization in [`3`](#3-device-initialization). The driver
69//! MUST NOT set a device status to MAGIC value. The driver MUST fill the
70//! necessary fields of the ring buffer configuration space
71//! (size, queue_addr_low and queue_addr_high) after driver sets the DRIVER_OK
72//! status bit.
73//!
74//! #### 2.1.2 Device Requirements: Device Status Field
75//! The device MUST initialize device status to 0x74726976 upon start and reset.
76//!
77//! The driver MUST enable READY after it validate the queue address and size
78//! negotiated with the virtio driver.
79//! The device SHOULD set RESET when it enters an error state that a reset is
80//! needed.
81//!
82//! #### 2.2 Ring buffer
83//! The mechanisms for data transport on svirtb is ring buffer.
84//! Driver makes requests available to device by enqueue an entry or multiple
85//! entries to the ring buffer and write the index to the doorbell
86//! (ring_buffer_head). Device executes the requests and, when complete, update
87//! the lastly consume index to the ring_buffer_tail.
88//!
89//! #### 2.2.1 Ring buffer entry
90//! Each entry of the ring buffer is consists of the four parts.
91//! ```C
92//! struct VirtQueueEntry {
93//!     le64 addr;
94//!     le64 size;
95//!     le64 sector;
96//!     le32 cmd;
97//! }
98//! ```
99//! * addr: Physical address of the buffer
100//! * size: size of the buffer
101//! * sector: sector of the vritual disk
102//! * cmd: indicates the command. 0 is read, and 1 is write.
103//!
104//! ### 3. Device Initialization
105//! The driver MUST follow this sequence to initialize a device:
106//! 1. Check the magic exists in status field.
107//! 2. Write the DRIVER_OK to the status field.
108//! 3. Check whether status field is still DRIVER_OK.
109//! 4. Perform ring buffer configuration; reading and writing device's ring
110//!    buffer configuration space.
111//! 5. Set the status field to READY.
112//! 6. Check whether status field is READY. At this point the device is "live".
113//!
114//! The driver MUST NOT send any buffer available notifications to the device
115//! before setting READY.
116//!
117//! ### 4. virtio over mmio
118//! For simplicity of the implementation, the mmio region of the simple virtIO
119//! block always located on the 0xcafe0000.
120//!
121//! The layout of the mmio area is follow:
122//! ```C
123//! struct svirtb {
124//!     le32 status;
125//!     le32 queue_size;
126//!     le32 queue_addr_hi;
127//!     le32 queue_addr_lo;
128//!     le32 queue_head;
129//!     le32 queue_tail;
130//! }
131//! ```
132//! * status: Device status bits. Reading from this register returns the current
133//!   device status flags. Initialized with magic by device - 0x74726976 (a
134//!   Little Endian equivalent of the 'virt' string).
135//! * queue_size: Writing to this register notifies the device what size of the
136//!   queue the driver will use (select the length of the queue).
137//! * queue_addr_hi: high part of ring buffer physical address (32-63 bits)
138//! * queue_addr_lo: low part of ring buffer physical address (0-31 bits)
139//! * queue_head: head index of the ring buffer
140//! * queue_tail: tail index of the ring buffer
141//!

146//! [`VirtQueueEntry`] through an struct called [`VirtQueueFetcher`].
147//! You can utilize [`VirtQueueFetcher`] to implement this project.
148//!

150//! [`VirtQueueEntry`]: crate::virtio::virt_queue::VirtQueueEntry
151//! [`VirtQueueFetcher`]: crate::virtio::virt_queue::VirtQueueFetcher
