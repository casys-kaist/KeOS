1//! Project 1: VMCS and VMExits.
2//!
3//! In the first project, you will implement the basic building blocks of the
4//! virtualization. Although the hardware-based virtualization gives a safe and
5//! convinient way to support virtualize the machine, it is not a perfect tool.
6//!
7//! Some less-frequently used and safety sensitive operations are required to be
8//! trap-and-emulated. Examples of such sensitive instructions include cpuid,
9//! port-mapped I/O, and memory-mmaped I/O.
10//!
11//! Some operations have high cost to trap-and-emulate multiple steps, they are
12//! optimization that combines multiple operations into a single operation
13//! called hypercall.
14//!
15//! You will learn about multiple trap-and-emulate operations by implementing a
16//! simple print function in three different ways: hypercall, port-mapped I/O
17//! and memory-mapped I/O. In project2, you will play on first two, hypercall
18//! and port-mapped I/O.
19//!
20//! Also, you will learn the concepts of instruction emulation.
21//! Some instructions (e.g. cpuid, rdmsr, wrmsr) are sensitive to run directly
22//! on guest cpu or required some modifications. The VMM requires to
23//! trap-and-emulate those instructions.
24//!
25//! While hands-on the project, you will become familiar with kev's
26//! infrastructure and the concepts of VMCS and VMExit.
27//!
28//! ## Background
29//! ### Virtual machine
30//! A virtual machine is an physical computer system that provides virtualized
31//! hardware resources. The virtual resources, such as CPU, memory, storage, and
32//! network interfaces, are presented to guest operating systems and
33//! applications as if they were physically present. By running on top of a
34//! virtual machine, normal programs and operating systems can be executed using
35//! these virtual resources.
36//!
37//! Multiple virtual machines can be operated on a single physical machine,
38//! and the distribution of physical resources to virtual machines is managed by
39//! a software layer called a Virtual Machine Monitor (VMM), also known as a
40//! Hypervisor. The VMM provides a layer of abstraction between the virtual
41//! machines and the underlying physical hardware, enabling physical resources
42//! to be shared among all VMs. Each VM environment is isolated from other VMs,
43//! so they do not know about each other and are unaware that they are
44//! controlled by the VMM.
45//!
46//! Modern CPU architectures, such as those from Intel and AMD, provide hardware
47//! virtualization support that enables system engineers to design custom
48//! virtual machine environments. By utilizing hardware-assisted virtualization,
49//! guest virtual machines can securely and efficiently access physical
50//! resources provided by the host machine. In the KeV projects, we will
51//! utilizes the Intel's hardware virtualization, called Intel VM-eXtension
52//! (Intel VMX).
53//!
54//! ### VM Exit
55//! Hardware virtualization is achieved by dividing the execution context into
56//! two modes: host (VMX-root mode in Intel VMX) and guest (VMX-non-root mode in
57//! Intel VMX). The VM Exit and VM Entry events facilitate mode switching
58//! between these two contexts. When a virtual machine operation is executed in
59//! non-root mode, certain sensitive instructions or events may trigger a VM
60//! exit, which transfers control from the guest os context to the host os
61//! context.
62//!
63//! When a VM exit occurs, the VMM takes control and decides how to handle the
64//! sensitive operation or event. The VMM may emulate the operation, modify the
65//! operation to suit the virtualized environment, or defer the operation to the
66//! underlying hardware. After the operation is handled, the VMM can choose to
67//! resume execution in the non-root context.
68//!
69//! ### Virtual Machine Control data Structure (Intel VMX)
70//! Virtual machine control data structures (VMCS) are structures used by x86
71//! during virtual machine execution. VMCS stores detailed information on how a
72//! virtual machine will operate and its internal states, including its CPU and
73//! memory, IO configurations, interrupt handling, and other hardware settings.
74//!
75//! Virtual machine monitor (VMM) can control and read hardware features by
76//! assigning difference VMCS to each virtual machine virtual processors.
77//! For instance, a virtual machine with four virtual cores would use four VMCS
78//! to control each core's functionality.
79//!
80//! To provide VMM with an abstract specification of the VMCS, which has
81//! difference data structures across processor vendors and versions, hypervisor
82//! access to VMCS is made through special instructions called `vmwrite` and
83//! `vmread`. `vmwrite` writes to a specific field of the activated VMCS and
84//! `vmread` reads a specific field of the activated VMCS. Therefore, x86
85//! provides ways to access and distinguish which VMCS is currently being set by
86//! the hypervisor. To this end, the VMCS states of Active, Current, and Clear
87//! have been added. In this project, hardware and software details for VMCS
88//! execution states and its associated instructions are hided from you.
89//!
90//! ## KeV
91//! KeV supplies a layer to abstraction to access the hardware-specific details.
92//! It provides abstractions of Virtual Machine ([`Vm`]) and virtual CPU
93//! ([`VCpu`]).
94//!
95//! ### Vm
96//! The [`Vm`] is an abstraction of a virtual machine. It contains multiple
97//! virtual CPUs and its internal states. You can interact with the [`Vm`]
98//! instance with the trait [`VmOps`], which defines the mutiple operations of a
99//! Virtual Machine. See the [`VmOps`] for list of the operations.
100//!
101//! ### VCpu
102//! The [`VCpu`] is an abstraction of a single virtual CPU. It holds the states
103//! to run virtual CPU, including VMCS, general purpose registers, virtual CPU
104//! ID, and VM Exit handlers. Each [`VCpu`] is an entity of scheudling. That is,
105//! it is a basically a thread of a host operating system!
106//!
107//! The VCpu thread runs in a loop that launch the VCpu, and handle the VM Exit.
108//! In the loop, the thread launches the guest operating system through
109//! `vmlaunch` or `vmresume`. After that, guest os traps back to the vmm by VM
110//! Exit, it resolves the reason of the VM Exit, handles the VM Exit, and
111//! returns back to the guest operating system. You can found the details of the
112//! loop on [`Vm::vcpu_thread_work`].
113//!
114//! ### Guest Memory Abstraction
115//! With the introduction of the guest operating system, KeV brings two terms:
116//! guest physical address ([`Gpa`]) and guest virtual addresss ([`Gva`]).
117//!
118//! In project 2, all you need to mind is that there is no guest physical
119//! address and guest virtual address is same as a host virtual address because
120//! we are not introduce the memory virtualization yet. We will revisit this
121//! topics in project 3 with memory virtualiation.
122//!
123//! ### VmexitController
124//! The [`VmexitController`] is the core interface to play with the VM Exits.
125//! It is a trait define as follow:
126//! ```rust
127//! pub trait VmexitController {
128//!    /// Handle the vmexit on this controller.
129//!    ///
130//!    /// Returns [`VmError::HandleVmexitFailed`] when failed to handle vmexit on this controller.
131//!    fn handle<P: Probe>(
132//!        &mut self,
133//!        reason: ExitReason,
134//!        p: &mut P,
135//!        generic_vcpu_state: &mut GenericVCpuState,
136//!    ) -> Result<VmexitResult, VmError>;
137//! }
138//! ```
139//! The VmexitController trait requires four arguments as input parameters
140//! * &mut self: Contains the self object of the given controller structure. You
141//!   can holds controller-specific states in `self`.
142//! * reason: Argument represents the [`ExitReason`] that holds the exit reasons
143//!   for the current vm exit. For instance, if the current vcpu is a hypercall,
144//!   the exit reasons would be [`BasicExitReason::Vmcall`].
145//! * p: [`Probe`] object that holds an information of memory state of the guest
146//!   operating system. You can use this object to translate address between
147//!   guest and host operating system.
148//! * generic_vcpu_state: Defines the states for the current vcpu. These states
149//!   comprise the current Virtual Machine Control Structure (VMCS), general
150//!   purpose registers (gen), and other relevant components which includes the
151//!   [`VmOps`]. See the [`GenericVCpuState`].
152//!
153//!
154//! #### ActiveVmcs
155//! [`ActiveVmcs`] is an abstraction to communicate with the activated VMCS.
156//! You can read or write to the activated VMCS with this struct.
157//! This also defines multiple helper functions to modify the VMCS.
158//!
159//! For instance,
160//! ```rust
161//! /// Forward to the next instruction.
162//! pub fn forward_rip(&self) -> Result<(), VmError> {
163//!    self.write(
164//!        Field::GuestRip,
165//!        self.read(Field::GuestRip)? + self.read(Field::VmexitInstructionLength)?,
166//!    )
167//! }
168//! ```
169//! The VMCS in this code (&self) is capable of reading its internal fields to
170//! read and write the RIP information necesasry for RIP forwarding. As like in
171//! this example, you can read and write the VMCS structure for fetching
172//! hardware informations of virtual machine in your KeV projects.
