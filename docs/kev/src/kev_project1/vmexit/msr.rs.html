1//! Model-specific register vmexit controller.
2//!
3//! x86 processors include a collection of specialized registers that permit the
4//! customization of CPU capabilities. Each register possesses a 64-bit space,
5//! and accessing these registers necessitates using `rdmsr` (read) or `wrmsr`
6//! (write) instructions.
7//!
8//! The `rdmsr` instruction reads a register that is determined by the index
9//! value provided in the ECX register. The retrieved MSR value is stored in the
10//! EDX and EAX registers, with the high-order 32 bits of the MSR value placed
11//! in the EDX register, and the low-order 32 bits placed in the EAX register.
12//! In 64-bit architectures, the high-order 32 bits of both the RDX and RAX
13//! registers are set to 0.
14//!
15//! The `wrmsr` instruction writes a value to a register specified in the ECX
16//! register as an index. The value written to the MSR is obtained from the EDX
17//! and EAX registers. Specifically, the value in EDX is written to MSR\[63:32\]
18//! (high-order 32bits), and the value in EAX is written to MSR\[31:0\]
19//! (low-order 32bits).
20//!
21//! When managing virtual machines, the default behavior is for the host to
22//! intercept `rdmsr`/`wrmsr` instructions from the guest. Therefore, the host
23//! needs to emulate these MSR access requests.
24//!
25//! ## Tasks
26//! In this part, you requires to write a manager to maintain the msr. When
27//! configuring the VCpu, the msr handlers are registered via
28//! [`Controller::insert`]. After that, when the guest operating system is
29//! trapped back to the VMM by executing either `rdmsr` or `wrmsr`, the control
30//! passed to the [`Controller::handle`]. In the function, the handler finds the
31//! corresponding msr handlers, runs the handler and reflect the result into the
32//! VCpu state. Again, you **MUST** forward the vCPU instruction pointer (rip)
33//! to prevent it from executing the same instructions infinitely.
