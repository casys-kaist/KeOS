39//! ## Why Rust?
40//!
41//! We have chosen **Rust** for this project because of its **memory safety**,
42//! **zero-cost abstractions**, and most importantly, its **concurrency model**.
43//! These features make Rust an excellent choice for **operating system
44//! development**.
45//!
46//! In traditional system programming languages, concurrency and memory bugs
47//! such as data races, use-after-free errors, and null pointer dereferences are
48//! common. Rust prevents these issues at compile time by enforcing strict
49//! **ownership, borrowing, and lifetime rules**. This allows you to write
50//! **safe and efficient concurrent code** without sacrificing performance, and
51//! **reduces debugging time** for those bugs.
52//!
53//! By using Rust in **KeOS**, you will:
54//! - **Develop safe and efficient concurrent programs** without the risk of
55//!   data races.
56//! - **Avoid common concurrency pitfalls** such as race conditions.

58//! ## Project Structure
59//!
60//! The KeOS project is divided into five projects:
61//!
62//! 1. **[`System Call`]** – Learn how the OS interacts with user applications.
63//! 2. **[`Memory Management`]** – Implement basic memory management and
64//!    user-space execution.
65//! 3. **[`Advanced Memory Management`]** – Expand the KeOS's memory management
66//!    system with advanced features.
67//! 4. **[`Process Management`]** – Implement the advanced process management,
68//!    including round robin scheduler and sychronization primitives.
69//! 5. **[`File System`]** – Develop a simple yet functional filesystem for data
70//!    storage.
71//!
72//! Each project builds upon the previous ones, helping you progressively
73//! develop a deeper understanding of OS design.
74//!
75//!
76//! ## Implementation Notes
77//!
78//! In **KeOS**, each process/thread is assigned a fixed execution stack of
79//! `STACK_SIZE` bytes. While KeOS attempts to detect stack overflows, its
80//! detection is not perfect. **A stack overflow may lead to mysterious kernel
81//! panics.** To avoid this:
82//! - **Avoid declaring large data structures on the stack.**
83//! ```rust
84//! let v: [u8; 0x200000]; // ERROR: This may cause a stack overflow
85//! ```
86//!
87//! - **Instead, allocate large data structures on the heap using `Box`.**
88//! ```rust
89//! let v = Box::new([0u8; 0x200000]); // OK: Allocates on the heap
90//! ```
91//!
92//! ## Implementation Strategy
93//!
94//! We recommend using a **"TODO-driven" approach** to build KeOS
95//! systematically. This method ensures an **incremental and structured**
96//! development process:
97//!
98//! 1. **Run the code** and identify `todo!()` placeholders that cause panics.
99//! 2. **Implement the missing functionality**, ensuring it aligns with the
100//!    expected behavior described in the project requirements.
101//! 3. **Repeat** steps 1 and 2 until all test cases pass and the system behaves
102//!    correctly.
103//!
104//! This approach allows you to build your OS **one step at a time**,
105//! making debugging and understanding the system easier.

161//! ## Debugging with GDB
162//!
163//! KeOS supports debugging with **GDB** and **QEMU**. This section provides
164//! step-by-step instructions on how to set up and use GDB for effective
165//! debugging.
166//!
167//! ### Running GDB
168//!
169//! To launch KeOS in debug mode, run the following command **inside each grader
170//! directory**:
171//! ```bash
172//! $ GDB=1 cargo run <TESTCASE>
173//! ```
174//! You must specify a single test case to debug with a GDB.
175//!
176//! This starts **QEMU** and waits for a GDB connection on TCP **port 1234**.
177//! A `.gdbinit` script will also be generated to automate the debugging setup.
178//!
179//! In a **separate terminal**, start GDB using:
180//! ```bash
181//! $ rust-gdb keos_kernel
182//! ```
183//!
184//! **Why `rust-gdb`?**
185//! We recommend `rust-gdb`, as it provides better support for Rust-specific
186//! data structures and improves debugging readability.
187//!
188//! #### One-time setup
189//! Before using `rust-gdb`, you may need to modify your **`~/.gdbinit`** file
190//! to allow script execution. Add the following line:
191//! ```bash
192//! set auto-load safe-path /
193//! ```
194//!
195//! After launching `rust-gdb`, the execution will halt at the startup stage,
196//! showing output similar to this:
197//! ```bash
198//! $ rust-gdb
199//! warning: No executable has been specified and target does not support
200//! determining executable automatically. Try using the "file" command.
201//! 0x000000000000fff0 in ?? ()
202//! (gdb)
203//! ```
204//!
205//! Now, you can continue execution by typing:
206//!
207//! ### Inspect Each Core
208//!
209//! In **QEMU**, each **CPU core** is treated as a **separate thread**.
210//! When debugging multi-core execution, be aware that **some cores may panic
211//! while others continue running.**
212//!
213//! To inspect all active cores, use:
214//! ```bash
215//! (gdb) info threads
216//! ```
217//!
218//! This will display the state of each thread, including which CPU core it
219//! belongs to and its current stack frame. For example:
220//! ```text
221//! (gdb) info threads
222//! Id   Target Id         Frame
223//! * 1    Thread 1 (CPU#0 [running]) 0x000000000000fff0 in ?? ()
224//! 2    Thread 2 (CPU#1 [running]) 0x000000000000fff0 in ?? ()
225//! 3    Thread 3 (CPU#2 [running]) 0x000000000000fff0 in ?? ()
226//! 4    Thread 4 (CPU#3 [running]) 0x000000000000fff0 in ?? ()
227//! ```
228//!
229//! To switch to a specific **CPU core (thread)**, use:
230//! ```bash
231//! (gdb) thread {thread_id}
232//! ```
233//!
234//! This allows you to inspect registers, call stacks, and execution state
235//! per core.
236//!
237//! ---
238//!
239//! ## Analyzing Execution State
240//!
241//! ### Viewing the Call Stack (Backtrace)
242//!
243//! Use `backtrace` (or `bt`) to display the **call stack** of the current
244//! thread:
245//! ```bash
246//! (gdb) bt
247//! ```
248//!
249//! Each function call in the stack is represented as a **frame**.
250//! To switch to a specific frame, use:
251//! ```bash
252//! (gdb) frame {frame_id}
253//! ```
254//!
255//! Once inside a frame, you can inspect variables:
256//! ```bash
257//! (gdb) info args
258//! (gdb) info locals
259//! (gdb) i r
260//! ```
261//!
262//! **Debugging a Panic:**
263//! If you encounter a **kernel panic** during a test, use:
264//! 1. `info threads` to locate the crashing core
265//! 2. `bt` to examine the backtrace
266//! 3. `frame {frame_id}` to inspect function parameters
267//!
268//! ---
269//!
270//! ## Setting Breakpoints
271//!
272//! Breakpoints help stop execution at specific points. However, in **multi-core
273//! debugging**, regular breakpoints may not always work correctly.
274//! Instead, use **hardware breakpoints**:
275//! ```bash
276//! (gdb) hb * {address_of_breakpoint}
277//! ```
278//!
279//! To view the source code that the current CPU is executing, use:
280//! ```bash
281//! (gdb) layout asm
282//! (gdb) layout src
283//! ```
284//!
285//! ### Examples
286//!
287//! Here are some examples of how to set breakpoints in GDB:
288//! ```text
289//! (gdb) hbreak function_name    # Example: (gdb) hbreak Rounrobin::new
290//! (gdb) hbreak *address         # Example: (gdb) hbreak *0x1000
291//! (gdb) hbreak (file:)line      # Example: (gdb) hbreak rr.rs:95
292//! ```
293//!
294//! #### Example 1
295//!
296//! To debug the `check_affinity` test case in project 1, and set a breakpoint
297//! at `main.rs:115`, use:
298//! ```bash
299//! (gdb) hbreak main.rs:115
300//! ```
301//!
302//! Alternatively, you can set a breakpoint on the closure entry:
303//! ```bash
304//! (gdb) hbreak check_affinity::{{closure}}
305//! ```
306//!
307//! To limit debugging to one core, use `thread apply`:
308//! ```bash
309//! (gdb) thread apply 1 hbreak main.rs:115
310//! (gdb) c
311//! ```
312//!
313//! #### Example 2
314//!
315//! To stop at a breakpoint only when a specific condition is met (e.g., when a
316//! parameter is `0xcafe0000`), use:
317//! ```bash
318//! (gdb) hbreak walk if va.__0 == 0xcafe0000
319//! ```
320//!
321//! This approach allows you to focus on specific conditions and skip over
322//! unrelated calls.

370/// Enum representing errors that can occur during a kernel operation.
371///
372/// This enum is used to categorize errors encountered by the kernel operation.
373/// Each variant corresponds to a specific type of error that might
374/// occur during the handling of a kernel operation. These errors can be
375/// returned to the user program to indicate the nature of the failure.
376#[derive(Debug, Eq, PartialEq)]
377pub enum KernelError {
378    /// Operation is not permitted. (EPERM)
379    OperationNotPermitted,
380    /// No such file or directory. (ENOENT)
381    NoSuchEntry,
382    /// IO Error. (EIO)
383    IOError,
384    /// Exec format error. (ENOEXEC)
385    NoExec,
386    /// BAD file descriptor. (EBADF)
387    BadFileDescriptor,
388    /// Out of memory. (ENOMEM)
389    NoMemory,
390    /// Permission denied. (EACCES)
391    InvalidAccess,
392    /// Bad address. (EFAULT)
393    BadAddress,
394    /// Device or resource busy. (EBUSY)
395    Busy,
396    /// File exists. (EEXIST)
397    FileExist,
398    /// Not a directory. (ENOTDIR)
399    NotDirectory,
400    /// Is a directory. (EISDIR)
401    IsDirectory,
402    /// Invalid arguement. (EINVAL)
403    InvalidArgument,
404    /// Too many open files. (EMFILE)
405    TooManyOpenFile,
406    /// No space left on device. (ENOSPC)
407    NoSpace,
408    /// Broken pipe. (EPIPE)
409    BrokenPipe,
410    /// File name too long. (ENAMETOOLONG)
411    NameTooLong,
412    /// Invalid system call number. (ENOSYS)
413    NoSuchSyscall,
414    /// Directory not empty (ENOTEMPTY)
415    DirectoryNotEmpty,
416    /// File system is corrupted. (EFSCORRUPTED)
417    FilesystemCorrupted(&'static str),
418    /// Operation is not supported. (ENOTSUPP)
419    NotSupportedOperation,
420}
421
422impl KernelError {
423    /// Converts the [`KernelError`] enum into a corresponding `usize` error
424    /// code. The result is cast to `usize` for use as a return value in
425    /// system calls.
426    pub fn into_usize(self) -> usize {
427        (match self {
428            KernelError::OperationNotPermitted => -1isize,
429            KernelError::NoSuchEntry => -2,
430            KernelError::IOError => -5,
431            KernelError::NoExec => -8,
432            KernelError::BadFileDescriptor => -9,
433            KernelError::NoMemory => -12,
434            KernelError::InvalidAccess => -13,
435            KernelError::BadAddress => -14,
436            KernelError::Busy => -16,
437            KernelError::FileExist => -17,
438            KernelError::NotDirectory => -20,
439            KernelError::IsDirectory => -21,
440            KernelError::InvalidArgument => -22,
441            KernelError::TooManyOpenFile => -24,
442            KernelError::NoSpace => -28,
443            KernelError::BrokenPipe => -32,
444            KernelError::NameTooLong => -36,
445            KernelError::NoSuchSyscall => -38,
446            KernelError::DirectoryNotEmpty => -39,
447            KernelError::FilesystemCorrupted(_) => -117,
448            KernelError::NotSupportedOperation => -524,
449        }) as usize
450    }
451}

453/// The given `isize` does not indicate an [`KernelError`].
454#[derive(Debug, Eq, PartialEq)]
455pub struct TryFromError {
456    e: isize,
457}

459impl TryFrom<isize> for KernelError {
460    type Error = TryFromError;
461
462    // Required method
463    fn try_from(value: isize) -> Result<Self, Self::Error> {
464        match value {
465            -1 => Ok(Self::OperationNotPermitted),
466            -2 => Ok(Self::NoSuchEntry),
467            -5 => Ok(Self::IOError),
468            -8 => Ok(Self::NoExec),
469            -9 => Ok(Self::BadFileDescriptor),
470            -12 => Ok(Self::NoMemory),
471            -13 => Ok(Self::InvalidAccess),
472            -14 => Ok(Self::BadAddress),
473            -16 => Ok(Self::Busy),
474            -17 => Ok(Self::FileExist),
475            -20 => Ok(Self::NotDirectory),
476            -21 => Ok(Self::IsDirectory),
477            -22 => Ok(Self::InvalidArgument),
478            -24 => Ok(Self::TooManyOpenFile),
479            -28 => Ok(Self::NoSpace),
480            -32 => Ok(Self::BrokenPipe),
481            -36 => Ok(Self::NameTooLong),
482            -38 => Ok(Self::NoSuchSyscall),
483            -39 => Ok(Self::DirectoryNotEmpty),
484            -117 => Ok(Self::FilesystemCorrupted("")),
485            -524 => Ok(Self::NotSupportedOperation),
486            e => Err(TryFromError { e }),
487        }
488    }
489}

525/// The entry of the KeOS for bootstrap processor.
526#[unsafe(no_mangle)]
527#[allow(clippy::missing_safety_doc)]
528pub unsafe fn rust_main(core_id: usize, regions: abyss::boot::Regions, cmd: Option<&'static [u8]>) {
529    info!(
530        "\x1bc\n\
531__  __     ___  ____  \n\
532| |/ /___ / _ \\/ ___| \n\
533| ' // _ \\ | | \\___ \\ \n\
534| . \\  __/ |_| |___) |\n\
535|_|\\_\\___|\\___/|____/ \n\
536\n\
537KAIST educational Operating System\n\
538Copyright 2025 Computer Architecture and Systems Lab\n"
539    );
540    // Init memory.
541    unsafe {
542        info!("Memory: init memory.");
543        crate::mm::init_mm(regions);
544        if let Some(cmd) = cmd {
545            KERNEL_CMDLINE = CString::from_vec_with_nul(cmd.into()).ok();
546        }
547    }
548    info!("Devices: init pci devices.");
549    unsafe {
550        abyss::dev::pci::init();
551    }
552    // Load debug symbols
553    info!("Panicking: Load debug symbols.");
554    if !crate::lang::panicking::load_debug_infos() {
555        warning!("Failed to read kernel image. Disabling stack backtrace.");
556    }
557
558    unsafe extern "Rust" {
559        fn main(conf_builder: SystemConfigurationBuilder);
560    }
561    unsafe {
562        main(SystemConfigurationBuilder { _p: () });
563        abyss::boot::bootup_mps();
564        // Kill the kernel low address.
565        (abyss::addressing::Pa::new({
566            unsafe extern "C" {
567                static mut boot_pml4e: u64;
568            }
569            boot_pml4e as usize
570        })
571        .unwrap()
572        .into_kva()
573        .into_usize() as *mut mm::page_table::Pml4e)
574            .as_mut()
575            .unwrap()
576            .set_flags(mm::page_table::Pml4eFlags::empty());
577        println!(
578            "Command line: {}",
579            KERNEL_CMDLINE
580                .as_ref()
581                .and_then(|cmd| cmd.to_str().ok())
582                .unwrap_or("")
583        );
584    }
585
586    crate::interrupt::register(32, |_| scheduler().timer_tick());
587    crate::interrupt::register(126, mm::tlb::handler);
588    crate::interrupt::register(127, |_regs| { /* no-op */ });
589    BOOT_DONE.store(true, core::sync::atomic::Ordering::SeqCst);
590    // Now kernel is ready to serve task.
591    crate::thread::scheduler::idle(core_id);
592}

607// Test utilities
608#[doc(hidden)]
609pub trait TestCase
610where
611    Self: Sync + Send,
612{
613    fn name(&'static self) -> &'static str;
614    fn run(&'static self, task: Box<dyn Task>) -> bool;
615}
616
617impl<T> TestCase for T
618where
619    T: Fn() + Send + Sync + 'static,
620{
621    fn name(&'static self) -> &'static str {
622        core::any::type_name::<T>()
623    }
624    fn run(&'static self, task: Box<dyn Task>) -> bool {
625        print!("test {} ... ", core::any::type_name::<T>());
626        if crate::thread::ThreadBuilder::new(core::any::type_name::<T>())
627            .attach_task(task)
628            .spawn(self)
629            .join()
630            == 0
631        {
632            println!("ok");
633            true
634        } else {
635            println!("FAILED");
636            false
637        }
638    }
639}

641/// A driver for running tests.
642pub struct TestDriver<T: Task + Default + 'static> {
643    _t: core::marker::PhantomData<T>,
644}
645
646impl<T: Task + Default + 'static> TestDriver<T> {
647    /// Run the given tests.
648    pub fn start<const TC: usize>(tests: [&'static dyn TestCase; TC]) {
649        crate::thread::ThreadBuilder::new("test_main").spawn(move || {
650            let tests = match unsafe { KERNEL_CMDLINE.as_ref() } {
651                Some(cmd) if cmd.to_str() != Ok("") => {
652                    let filter = cmd
653                        .to_str()
654                        .expect("Failed to parse cmd")
655                        .split(" ")
656                        .collect::<BTreeSet<_>>();
657                    tests
658                        .iter()
659                        .filter(|test| {
660                            let name = test.name();
661                            let r = name.split("::").next().map(|n| n.len() + 2).unwrap_or(0);
662                            filter.contains(&name[r..])
663                        })
664                        .collect::<Vec<_>>()
665                }
666                _ => tests.iter().collect::<Vec<_>>(),
667            };
668            let (total, mut succ) = (tests.len(), 0);
669            println!(
670                "Running {} test{}",
671                total,
672                if total == 1 { "" } else { "s" }
673            );
674
675            for test in tests {
676                if test.run(Box::new(T::default())) {
677                    succ += 1;
678                }
679            }
680            println!(
681                "test result: {}. {} passed; {} failed",
682                if total == succ { "ok" } else { "FAILED" },
683                succ,
684                total - succ
685            );
686
687            unsafe {
688                abyss::x86_64::power_control::power_off();
689            }
690        });
691    }
